## 1. Goal

Deliver CodexLAG V1.1 as the first post-V1 expansion release with three concrete outcomes:

- OpenAI official accounts can sign in from inside the desktop app through a browser-based OAuth flow with a loopback callback and PKCE.
- CodexLAG can manage both the OpenAI official provider and a generic OpenAI-compatible provider through a shared provider registry instead of ad-hoc per-provider logic.
- The desktop UI grows from a solid V1 operations console into a more expressive operator surface with a richer capability matrix, stronger policy authoring, and deeper diagnostics.

V1.1 is not a platform rewrite. It is a bounded release that extends the existing V1 runtime, routing, persistence, and UI foundations without reopening already-closed V1 scope.

## 2. Confirmed Scope

### 2.1 V1.1 Must Deliver

V1.1 must satisfy all of the following:

- OpenAI official in-app login only
- browser launch from the desktop app using the default system browser
- local loopback callback for the OAuth authorization code return path
- PKCE-based authorization code flow
- persisted OpenAI session state with refresh-token support
- automatic token refresh on startup and before protected provider operations when refresh is possible
- explicit logout and re-authentication controls
- provider registry that supports:
  - `openai_official`
  - `generic_openai_compatible`
- generic provider configuration that supports:
  - `base_url`
  - credential storage
  - model discovery when available
  - manual model entry when discovery is unavailable or incomplete
- richer model-level capability matrix UI
- more expressive policy authoring UX than V1
- expanded diagnostics and operator-facing visibility beyond V1 release gates

### 2.2 V1.1 Explicitly Excludes

The following are out of scope for V1.1:

- additional official providers beyond OpenAI
- device-code login as a first-class auth path
- custom protocol callback as the primary auth return path
- remote multi-user account management
- hosted control plane behavior
- provider plugin loading from external binaries or remote manifests
- speculative future providers with bespoke auth before the OpenAI and generic contracts are stable

## 4. Architecture Direction

V1.1 should preserve the existing Rust-owned runtime and React/Tauri desktop split, but it needs a cleaner internal platform layer than V1 required.

### 4.1 Auth Runtime

Introduce a dedicated auth runtime for provider-backed desktop login.

This layer owns:

- auth session start
- PKCE generation
- CSRF/state validation
- loopback callback listener lifecycle
- token exchange
- refresh scheduling and refresh-on-demand
- logout and session invalidation

This logic should not live directly inside the UI-facing command handlers. Commands should call into a dedicated auth service.

### 4.2 Provider Registry

Introduce a provider registry modeled after the CLIProxyAPI registry pattern instead of continuing with scattered provider-specific branches.

The registry should register provider adapters by stable provider identifier and expose a shared contract for:

- auth capabilities
- credential/session status
- model discovery
- capability probing
- request execution
- balance support
- diagnostics metadata

V1.1 should register exactly two providers:

- `openai_official`
- `generic_openai_compatible`

### 4.3 Unified Account And Capability Inventory

The runtime should project provider-backed accounts and generic provider configs into one inventory layer used by:

- routing candidate construction
- account management UI
- capability matrix UI
- diagnostics

This prevents the UI and routing layers from learning provider-specific storage details.

### 4.4 Observability Layer

V1.1 should extend observability beyond request lifecycle rows.

It should add stable tracking for:

- auth session health
- token refresh outcomes
- capability probe results
- provider health summaries
- policy-driven candidate acceptance and rejection reasons
- operator-visible diagnostics snapshots

### 4.5 Shared Interfaces Locked Before Parallel Work

Before the auth, registry, inventory, UI, and diagnostics tracks split into parallel worktrees, V1.1 should freeze the cross-track contracts below in the design doc itself.

`ProviderSessionSummary` is the UI-facing session summary returned by provider auth flows and refresh operations:

```rust
pub struct ProviderSessionSummary {
    pub provider_id: String,
    pub account_id: String,
    pub display_name: String,
    pub auth_state: String,
    pub expires_at_ms: Option<i64>,
    pub last_refresh_at_ms: Option<i64>,
    pub last_refresh_error: Option<String>,
}
```

`ProviderAdapter` is the minimum stable registry contract shared by the official and generic OpenAI-compatible providers:

```rust
pub trait ProviderAdapter {
    fn provider_id(&self) -> &'static str;
    fn supports_browser_login(&self) -> bool;
    fn supports_balance(&self) -> bool;
}
```

`ModelCapabilitySummary` is the normalized capability row projected into routing, matrix, and diagnostics surfaces:

```rust
pub struct ModelCapabilitySummary {
    pub provider_id: String,
    pub account_id: String,
    pub model_id: String,
    pub supports_tools: bool,
    pub supports_streaming: bool,
    pub supports_reasoning: bool,
    pub source: String,
}
```

Any implementation that needs to extend one of these contracts should update the design doc and dependent subplans first rather than silently diverging across worktrees.

## 5. OpenAI Auth Design

### 5.1 Primary Flow

OpenAI in-app login should use:

- external browser launch
- authorization code flow
- PKCE
- loopback callback

This matches the desired desktop-user experience and aligns with the CLIProxyAPI OpenAI auth direction, where the auth URL is constructed with `response_type=code`, loopback redirect URI, PKCE challenge, and state.

### 5.2 Callback Strategy

Use a dedicated temporary loopback callback listener rather than reusing the existing Codex gateway host.

Reasoning:

- the gateway host and auth callback have different lifecycles
- auth callback state is short-lived and stateful
- keeping auth callback separate reduces accidental coupling between provider login and gateway request serving

The callback listener should:

- bind to `127.0.0.1`
- prefer an available ephemeral port within a bounded safe strategy
- shut down immediately after success, timeout, or cancellation

### 5.3 Session Persistence

OpenAI sessions should be persisted locally with a split-storage model:

- sensitive token material in the existing secret-storage layer
- non-secret session metadata in SQLite

Persisted metadata should include:

- provider id
- account id / subject id
- display label
- granted scopes
- issued-at / expires-at timestamps
- refresh status metadata
- last refresh attempt outcome
- auth health status

### 5.4 Refresh Semantics

V1.1 should support:

- startup refresh when a session is near expiry or expired but refreshable
- just-in-time refresh before provider operations that require a valid access token
- explicit refresh diagnostics surfaced to the UI

If refresh fails, the account should become visibly degraded rather than silently disappearing.

## 6. Generic OpenAI-Compatible Provider Design

### 6.1 Registry-Oriented Design

The generic provider should be implemented as a first-class registry entry, not as a one-off relay special case.

Its configuration should include:

- provider id
- user-facing label
- base URL
- credential reference
- optional organization or project headers when needed
- discovery mode
- manual model entries

### 6.2 Capability Strategy

The generic provider should support two model/capability sources:

- discovery from provider endpoints when available
- operator-managed manual model entries when discovery is absent or incomplete

The runtime should normalize both sources into the same capability inventory format used by the OpenAI official provider.

## 7. UI Design Direction

### 7.1 Server-State Management

The frontend should adopt `@tanstack/react-query` for provider sessions, capability inventory, policy data, and diagnostics state.

Reasoning:

- V1.1 introduces more independent async surfaces than V1
- query invalidation after login, logout, refresh, probe, save, or reconnect becomes routine
- this is better handled by a server-state library than additional bespoke state wiring

### 7.2 Capability Matrix

The richer capability matrix should move to a table-based surface built on `@tanstack/react-table`.

The table should support:

- sorting
- filtering
- column visibility
- provider/account scoping
- model capability comparison
- degraded/expired auth visibility

### 7.3 Form And Contract Validation

Use `zod` for:

- provider config validation
- auth callback payload validation
- policy editor validation
- diagnostics filter validation

The same schema family should define both UI-side validation and command payload validation where practical.

### 7.4 Policy Authoring

V1.1 policy authoring should move beyond minimum-form editing and add:

- clearer selection-order editing
- candidate preview where possible
- validation feedback before save
- stronger visibility into fallback and retry outcomes

### 7.5 Diagnostics

The diagnostics surface should grow into a proper operator tool, not just a release-gate aid.

It should expose:

- auth session health
- recent refresh attempts
- provider capability probe outcomes
- request/routing failure summaries
- candidate rejection reasons
- key runtime paths and data locations that matter operationally

## 8. Recommended Third-Party Libraries

V1.1 should deliberately adopt mature libraries where they remove custom infrastructure work.

### 8.1 Rust / Tauri

- `openidconnect-rs`
  - use for authorization code flow with PKCE, discovery-compatible configuration where applicable, nonce/state handling, and token exchange structure
- `reqwest`
  - continue to use for provider HTTP interactions, refresh, and capability probes
- `@tauri-apps/plugin-opener`
  - use to open the system browser for login initiation instead of custom shell-opening code

### 8.2 React

- `@tanstack/react-query`
  - use for query/mutation/invalidation flows around auth sessions, provider inventory, capability probes, and diagnostics
- `@tanstack/react-table`
  - use for the capability matrix and diagnostics tables
- `zod`
  - use for command payload and form validation

These libraries are preferred because they reduce custom state-management, table, and validation code that would otherwise slow delivery and increase defect risk.

## 9. Parallel Execution Model

V1.1 should be planned for parallel execution by default.

### 9.1 Worktree Requirement

Implementation should use isolated git worktrees rather than one shared working directory.

Preferred structure:

- `.worktrees/v1_1_auth`
- `.worktrees/v1_1_registry`
- `.worktrees/v1_1_ui`
- `.worktrees/v1_1_diagnostics`

Before creating project-local worktrees, the implementation workflow must verify the directory is ignored by git.

### 9.2 Parallelism Rule

Parallel work should begin only after:

- the V1 archive move is complete
- the V1.1 master spec and subplans are written
- shared interfaces for auth session state, provider registry contracts, and capability inventory are locked

After that point, the implementation can split into parallel tracks:

- OpenAI auth foundation
- provider registry and generic provider support
- frontend data-layer and matrix scaffolding
- diagnostics UX and policy-authoring enhancements

### 9.3 Commit Cadence

Each track should produce its own reviewable commits. Do not hold V1.1 changes for one giant final commit.

## 10. Workstream Decomposition

V1.1 should be implemented through one master plan plus the following subplans:

- `v1.1-archive-and-doc-structure`
- `v1.1-openai-auth`
- `v1.1-provider-registry`
- `v1.1-account-and-capability-inventory`
- `v1.1-capability-matrix-ui`
- `v1.1-policy-authoring`
- `v1.1-diagnostics`
- `v1.1-release-gates`

The master plan should define dependencies, integration checkpoints, and release criteria, but detailed code/test steps should live in the subplans.

## 11. Testing And Release Gates

V1.1 release gates should keep V1 coverage and add:

- Rust tests for OpenAI auth state handling, callback validation, session persistence, refresh, and logout
- Rust tests for provider registry behavior and generic provider normalization
- frontend tests for login state, session degradation, capability matrix interactions, policy validation, and diagnostics flows
- integration tests for end-to-end OpenAI login orchestration with mocked provider endpoints where needed

V1.1 must not weaken the existing V1 release verification.
