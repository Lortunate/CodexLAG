## 1. Goal

Deliver CodexLAG V1.2 as the first post-V1.1 desktop maturity release with three concrete outcomes:

- CodexLAG grows from a single-official-provider desktop app into a bounded multi-provider desktop console aligned with the existing CLIProxyAPI provider baseline.
- The desktop auth runtime expands beyond the V1.1 OpenAI-only browser-PKCE path and adds first-class provider auth profiles only for the account login methods already evidenced in CLIProxyAPI: browser OAuth with callback handling and API-key-backed providers.
- The operator surface becomes meaningfully multi-provider: account onboarding, capability visibility, policy preview, request explainability, and diagnostics all need to remain coherent when different providers use different auth shapes.

V1.2 is still a single-machine desktop release. It continues the local-first product line rather than introducing any hosted control-plane or multi-user behavior.

## 2. Confirmed Scope

### 2.1 V1.2 Must Deliver

V1.2 must satisfy all of the following:

- additional official providers beyond OpenAI, bounded to the CLIProxyAPI-aligned desktop provider set
- first-class desktop auth profiles:
  - `browser_oauth_pkce`
  - `static_api_key`
- browser-based providers continue to use local callback handling compatible with the current CLIProxyAPI-aligned OAuth shape
- provider registry expansion that can register and describe at least:
  - `openai_official`
  - `claude_official`
  - `gemini_official`
- auth/runtime contracts that describe auth profile kind, login affordances, degraded session state, and provider-specific recovery guidance
- unified inventory projection across browser-session accounts, device-code accounts, and API-key-backed accounts
- provider-aware account onboarding UX that explains:
  - which auth path the provider uses
  - what secrets or browser steps are required
  - what the current auth health state is
- richer policy and diagnostics surfaces for a multi-provider runtime, including:
  - provider-scoped capability comparison
  - provider-specific candidate rejection reasons
  - request explainability for final route choice
  - degraded account visibility that does not silently hide partially broken providers
- release gates that prove multi-provider routing, persistence, auth, and UI behavior under local desktop conditions

### 2.2 V1.2 Explicitly Excludes

The following remain out of scope for V1.2:

- remote multi-user account management
- hosted control plane behavior
- provider plugin loading from external binaries or remote manifests
- arbitrary provider expansion with no CLIProxyAPI baseline
- sync across multiple desktop machines
- cloud-hosted audit storage
- mobile-specific auth UX

## 3. Why V1.2 Follows V1.1

The existing documentation already points to this release direction:

- V1.1 explicitly excluded additional official providers and other auth-path expansion beyond the initial browser-based OpenAI login.
- The archived V1 product/completion docs consistently treated CLIProxyAPI alignment as the provider-behavior baseline.
- The foundation spec still requires local runtime closure for gateway, provider, routing, and persistence behavior.

That makes V1.2 the correct place to complete the next bounded layer of desktop capability without reopening the non-goals around remote control planes or plugin ecosystems.

## 4. Architecture Direction

### 4.1 Desktop Auth Profiles

V1.2 should stop treating provider auth as a provider-specific one-off and instead introduce a stable auth profile abstraction constrained to the account login shapes already present in CLIProxyAPI:

- `browser_oauth_pkce`
  - external browser launch
  - PKCE
  - state validation
  - local callback handling compatible with the existing OAuth callback flow
- `static_api_key`
  - no browser or callback lifecycle
  - local credential validation and health probing

Commands and UI flows should target auth profiles, not provider-specific ad-hoc branches.

### 4.2 Provider Registry V2

The registry contract should grow beyond the V1.1 minimum shape and describe:

- stable provider id
- auth profile kind
- supported login affordances
- model discovery strategy
- capability probe support
- diagnostics metadata
- credential/session validation strategy
- route-preparation hooks for upstream execution

V1.2 should keep the registry bounded to a fixed built-in set rather than turning it into a plugin system.

### 4.3 Account And Inventory Projection

The runtime should project all provider-backed accounts into one operator-facing inventory regardless of auth profile.

The projected layer should normalize:

- provider id
- account id
- display label
- auth profile
- auth health state
- last successful verification timestamp
- last failure summary
- model/capability rows
- provider-specific diagnostics attachments

This layer continues to shield routing, UI, and diagnostics from persistence-specific details.

### 4.4 Routing And Request Explainability

A multi-provider desktop runtime needs more than a candidate list.

V1.2 should expose operator-visible request explainability for:

- which provider/account candidates were considered
- which capability requirements filtered candidates out
- which auth-health checks degraded or blocked candidates
- why the final route was selected
- which upstream failure caused retry, fallback, or terminal stop

This is still a local operator tool, not a remote fleet-control feature.

### 4.5 Provider-Aware Operator Surface

The desktop UI should evolve from the V1.1 operator console into a provider-aware workbench:

- account onboarding should present the correct auth flow for each provider
- inventory should make auth profile and account health obvious
- capability matrix should support provider-family filtering and degraded-state visibility
- policy authoring should preview routing consequences across mixed provider families
- diagnostics should group information by provider, account, auth state, and request outcomes

## 5. Provider Scope

V1.2 should remain intentionally bounded.

### 5.1 Built-In Official Provider Set

The built-in official provider set for V1.2 should be:

- `openai_official`
- `claude_official`
- `gemini_official`

This matches the existing CLIProxyAPI provider vocabulary closely enough to preserve alignment while still keeping the desktop release manageable.

### 5.2 Generic Provider Handling

The V1.1 `generic_openai_compatible` path remains supported, but V1.2 does not broaden generic-provider scope into a plugin framework.

Generic providers stay subordinate to the built-in registry and local config model.

## 6. Auth And Callback Design

### 6.1 Browser Providers

Browser-based providers should support:

- PKCE-based auth
- explicit provider-owned state and nonce validation
- local callback handling consistent with the current CLIProxyAPI-aligned OAuth callback shape

### 6.2 API-Key Providers

API-key-backed providers should support:

- local secret entry and secure storage
- explicit validation/probe step
- degraded state and actionable recovery feedback
- capability discovery or manual configuration where discovery is unavailable

## 7. UI Design Direction

### 7.1 Account Onboarding

V1.2 account onboarding should be provider-aware rather than a provider-specific pile of forms.

It should provide:

- provider picker
- auth-profile-specific form or flow
- inline validation before launch/save
- post-auth verification state
- recovery guidance when verification fails

### 7.2 Capability And Policy Surfaces

The capability matrix and policy editor should remain built on the V1.1 foundations, but V1.2 should add:

- provider-family filtering
- auth-health overlays
- request explainability preview where possible
- stronger preview of retry and fallback outcomes across mixed provider inventories

### 7.3 Diagnostics

Diagnostics should become the local operator truth surface for provider runtime health.

It should expose:

- account auth state by provider
- recent device-code or browser-auth attempts
- key validation/probe outcomes
- provider-specific capability refresh results
- request-selection and rejection reasoning
- local data locations and remediation hints that matter operationally

## 8. Shared Contracts To Lock Before Implementation

V1.2 should lock the following UI/runtime-facing contracts in the design doc before implementation branches split.

`ProviderAuthProfile` describes the auth shape the UI and runtime should expect:

```rust
pub enum ProviderAuthProfile {
    BrowserOauthPkce,
    StaticApiKey,
}
```

`ProviderDescriptor` is the registry-facing provider summary:

```rust
pub struct ProviderDescriptor {
    pub provider_id: String,
    pub auth_profile: ProviderAuthProfile,
    pub supports_custom_protocol_callback: bool,
    pub supports_loopback_callback: bool,
    pub supports_model_discovery: bool,
    pub supports_capability_probe: bool,
}
```

`ProviderAccountHealth` is the operator-facing account/runtime summary:

```rust
pub struct ProviderAccountHealth {
    pub provider_id: String,
    pub account_id: String,
    pub auth_state: String,
    pub last_verified_at_ms: Option<i64>,
    pub last_error_code: Option<String>,
    pub last_error_message: Option<String>,
}
```

`RequestRouteExplanation` is the minimum explainability record projected into diagnostics and policy preview:

```rust
pub struct RequestRouteExplanation {
    pub request_id: String,
    pub selected_provider_id: Option<String>,
    pub selected_account_id: Option<String>,
    pub rejected_candidates: Vec<String>,
    pub rejection_reasons: Vec<String>,
}
```

## 9. Workstream Decomposition

V1.2 should be implemented through one master plan plus workstreams for:

- auth-profile runtime
- provider registry expansion
- inventory and health projection
- operator onboarding and account management UI
- policy and diagnostics explainability
- release gates

## 10. Testing And Release Gates

V1.2 release gates should keep V1 and V1.1 coverage and add:

- Rust tests for auth-profile state machines:
  - browser callback routing
  - device-code polling
  - API-key validation
- Rust tests for built-in provider registration and inventory normalization
- Rust integration tests for multi-provider route selection and explainability
- frontend tests for provider-aware onboarding, degraded account visibility, and diagnostics drilldown
- release workflow coverage for all new provider/auth-profile suites
